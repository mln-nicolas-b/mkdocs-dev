    Demander un accès à l'hyperviseur proxmox

    Pour demander un accès à cette plateforme, il suffit d'envoyer un mail à :

    • john.2.allen@atos.net
    • nicolas.becu@atos.net

    Formatez la demande de la façon suivante :

    • Nom: Nom de l'utilisateur à créer
    • Prenom: Prénom de l'utilisateur à créer
    • Email: Email de l'utilisateur à créer
    • Objet: Raison de la création de compte
    • ISO: Nom de l'ISO nécessaire
    • reseau: Besoin ou pas de réseau privé
    • projet: Projet associé à la demande

    Example :

    Example

    Nom: Nicolas  
Prenom: Becu  
Email: nicolas.becu@atos.net  
Objet: "Besoin d'un accès pour déployer un cluster kubernetes"  
ISO: talos-1.2.iso
reseau privé: oui
projet: casio

    Caution

    Le mot de passe est à changer dès réception de vos informations personnelles de connexion.
    La double authentification (TFA) est obligatoire. Les comptes qui en seront dépourvus seront désactivés.

    Wireguard

    La mise en place d'un tunnel VPN Wireguard nécessite l'échange de clé chifrée. Vous pouvez demander que l'on vous génère ces clés ou les générer vous même en suivant cette procédure.

    Dans le cas où vous générez votre propre paire de clés chiffrées pouvez récupérer le bout de fichier suivant pour configurer votre client.
    Quelques champs sont à compléter ou à modifier :

    • Adsress: Renseigner l'adresse IP fournie lors de la demande
    • PrivateKey: Renseigner la clé privée générée
    • PublicKey: s/cms7Td92Ywog2G8+789Qnudx0y5qImdvhkSaZEAhI=

    Il faudra également transmettre dans le mail la clé publique que vous avez générée.

    Info

    Dans le cas où vous de souhaitez pas générer votre propre paire de clé chiffrées, un fichier rempli vous sera transmis.

    [Interface]
Address = 10.0.1.x/32
PrivateKey = 
DNS = 10.0.0.1

[Peer]
PublicKey = 
Endpoint = rt.aosc-portal.com:51820
AllowedIPs = 0.0.0.0/0
PersistentKeepalive = 30

    Tip

    Pour télécharger le client spécifique à son système, rendez-vous sur la page de téléchargement officielle.
    Pour plus de détails, se référer à la documentation.

    Bonnes pratiques

    Lors de la création d'une machine virtuelle, il convient d'insérer dans les notes les informations suivantes pour aider le suivi automatisé de celle-ci :

    • owner: Propriétaire ou responsable de la machine virtuelle
    • creator: Personne à l'origine de la création de la machine virtuelle
    • creation: Date de création de la machine virtuelle
    • expiration: Date de fin d'utilité de la machine virtuelle
    • project: Projet lié à l'existance de la machine virtuelle
    • usage: Cas d'usage de la machine virtuelle

    Pour cela, il suffit d'ouvrir les notes de la vm et de renseigner les information au format yaml :

    Example

    Pour exemple, voici les notes renseignées pour la machine virtuelle PfSense :

    owner: Nicolas Becu
creator: Nicolas Becu
creation: 2022/09/01
expiration: no
project: sandbox
usage: Routeur logiciel pour la gestion du réseau de la plateforme

    Caution

    Toutes les machines virtuelles dépourvues de ces informations seront supprimées dans la semaine suivant leurs détections.

    Documentation

    La documentation est fournie par MkDocs. Chaque personnes ayant les droits requis sur le dépôt peut l'éditer pour ajouter ou mettre à jour des informations.

    Installation

    Installer MkDocs simplement avec le gestionnaire de paquet python :

    sudo pip3 install mkdocs

    Le thème de MkDocs est Material, il s'installe également avec le gestionnaire de paquet python :

    sudo pip3 install mkdocs-material

    Un plugin est actuellement en phase de test pour l'intégration de solution de schématisation.
    Étant dans la configuration de MkDocs, il est également à installer avec le gestionnaire de paquet python :

    sudo pip3 install mkdocs-kroki-plugin

    Configuration

    La configuration du wiki se trouve à la racine du dépôt, dans le fichier mkdocs.yaml.

    Si besoin, il est possible de rajouter de nouvelle entrées dans la navigation.
    Chaque entrée correspond à un menu de la navigation et chaque sous entrée correspond à une redirection d'url.

    nav:
  - Home: index.md
  - Proxmox:
    - 'Installation': 'proxmox/#installation'
    - 'Gestion des accès': 'proxmox/#gestion-des-acces'
    - 'Problèmes connus': 'proxmox/#problemes-connus'
    - 'Gestion de la virtualisation': 'proxmox/#gestion-de-la-virtualisation'
  - PfSense: 
    - 'OVH': 'pfsense/#ovh'
    - 'Gestion du réseau': 'pfsense/#gestion-du-reseau'
    - 'Gestion des accès': 'pfsense/#les-acces'
    - 'Wireguard': 'pfsense/#wireguard'
    - 'REST API': 'pfsense/#rest-api'

    Warning

    Ne pas toucher les configuration de plugins sans connaissances du système MkDocs !

    Usage

    Lors de l'édition locale de du wiki, il est particulièrement pratique de pouvoir visualiser le rendu des modifications opérées.
    Pour cela, il est possible de déployer le wiki localement :

    mkdocs serve

    Tip

    Ne pas oublier de commiter le code puis de le pousser sur la branch main pour mettre à disposition à tous les collaborateurs, la dernière version du wiki.

    Lorsque le résultat final est obtenu, il suffit le pousser sur la branch distante :

    mkdocs gh-deploy

    Info

    MkDocs gère seul le déploiement du wiki sur github.
    Le code va ainsi être transformer en site statique puis poussé sur une branch distinct.
    Les pages github pointent sur la racine de cette branche pour exposer le wiki.
