    PfSense

    OVH

    La machine virtuelle PfSense dispose d'un accès internet grâce à un routage interne au serveur dédié lui permettant d'être rattaché à une IP FailOver.

    Console web OVH

    Une fois connecter à la console web d'OVH, il faut se rendre dans la section bare metal cloud et dérouler l'onglet des serveurs dédiés pour pouvoir choisir celui désiré :

    Dans l'onglet de gestion des interfaces réseaux, on ouvre la console de gestion des adresses IP du serveur :

    On déplace l'adresse IP sur le serveur désiré :

    Pour finir, on ajoute une adresse MAC virtuelle à l'adresse IP, ce qui permettra de router le traffic réseau sur la machine virtuelle possédant cette adresse MAC.

    Configuration de la machine virtuelle

    On commence par configurer l'interface réseau vmbr0 du Proxmox. Celle doit être configurée en static sur l'adresse public du serveur.
    Sa passerelle réseau doit impérativement être configurée.

    Tip

    La configuration réseau de l'interface principale de Proxmox est effectuée par l'installeur.
    Il n'est normalement pas nécessaire d'y toucher.

    Info

    Les réseau OVH ont tous une passerelle en 254. Pour connaître sa passerelle, il suffit de prendre les 3 premiers octets de l'adresse du serveur, puis de compléter le dernier octet par 254.

    Ensuite, on ajoute à la machine virtuelle PfSense son interface WAN, en lui attribuant l'adresse MAC virtuelle de l'IP FailOver qui a été configurée au préalable.

    Dans l'interface web d'administration de pfsense, on configure la passerelle par défaut qui sera utilisée pour le routage de tout le traffic réseau entrant et sortant.

    Puis on paramètre PfSense pour pointer dessus.

    Dans l'interface web d'administration de pfsense, on configure l'interface réseau WAN.

    Routage interne

    Le routeur laisse passer sur son interface publique :

    • Le protocole ICMP : Pour permettre le ping du routeur à des fin surtout d'administration
    • Le protocole SSH sur le port 2222 : Pour permettre d'accéder en shell au routeur
    • Le protocole UDP sur le port de WIREGUARD (51820) : Pour permettre un accès VPN au uilisateurs

    Info

    Des règles d'accès pour les protocoles HTTP et HTTPS sont également en place mais désactivées par mesure de sécurité.
    Elle peuvent être réactivées sur demande si besoin est (maintenance, incident, etc.)

    Le réseau principale (default) est totalement ouvert sur tous le réseau interne géré par PfSense :

    Pour des besoins spécifiques, il est également possible de créer des routage restrictifs sur le traffic inter-réseau. Comme on peut le voir sur la capture suivante, le réseau peut communiquer avec le routeur ou avec internet.
    Il peut également communiquer avec le réseau lié à Wireguard (le VPN), mais aucun autre réseau, pas même le défault ne peuvent communiquer avec lui.

    Gestion des accès

    Note

    Définir une stratégie d'accès au PfSense

    Accès SSH

    Chaque compte créé dans le cadre d'un accès VPN OpenVPN se voit attribuer la possibilité de se connecter en shell sur le PfSense.
    Cela permet si besoin aux collaborateurs de faire du ProxyJump SSH si besoin (éviter d'initialiser une connexion VPN pour se connecter à une machine virtuelle).

    Accès web internes

    Les accès web interne se font via tunnel VPN.
    Pour PfSense, celui ci est accessible via le domain rt.infra.aosc, ou via toute les adresses passerelles de réseaux.

    Pour tout accès web à une un service hébergé sur une machine virtuelle, il suffit d'y accéder par son adresse IP.

    Accès web public

    Les accès web publics à PfSense sont désactivés au niveau pare feu.
    Ils peuvent être réactivés sur besoin en cas de problèmes réseaux.

    Pour les expositions de services herbergés sur la sandbox, il n'est à ce jour pas prévu d'exposer ces services (via reverse proxy par exemple).
    Les accès devront se faire par tunnel VPN.

    Wireguard

    PfSense fourni un accès VPN (Virtual Private Network) au réseau interne de l'hyperviseur Proxmox.

    Installation du client

    Wireguard est disponible sur énormément de plateformes.
    Pour les instructions d'installation, se référer à la documentation officielle

    Gestion des clés chriffrées

    Wireguard nécessite l'utilisation de clé chifrées privées et publiques.

    Créer une paire de clés chifrées :

    wg genkey | tee privatekey | wg pubkey > publickey

    Il est également possible de se rendre sur ce site, générer une configuration aléatoire pour récupérer la clé public et privée.

    Dans le cas où vous voudriez garder ces clés, il faudra les déplacer en lieu sûr.
    Dans le cas contraire, il suffit de les supprimer car plus utile.

    Configuration du client

    La méthode la plus simple est d'utiliser un fichier de configuration.

    • [Interface] : Configure la configuration locale
      • Address : Adresse ip (avec CIDR) de l'utilisateur
      • PrivateKey : Clé chifrée privée de l'utilisateur
      • DNS : Serveur DNS de référence un fois conncté.
    • [Peer] :
      • PublicKey : Clé chifrée publique du serveur distant
      • Endpoint : Adresse / Domaine du serveur distant avec son port (ex: ip:port)
      • AllowedIPs : Réseaux autorisés (Pour faire passer tous le traffic à travers la connexion: 0.0.0.0/0)
      • PersistentKeepalive : Persistence de la connexion en secondes

    Pour exemple, avec le fichier aosc.conf :

    [Interface]
Address = 10.0.1.x/32
PrivateKey = 
DNS = 10.0.0.1

[Peer]
PublicKey = 
Endpoint = rt.aosc-portal.com:51820
AllowedIPs = 0.0.0.0/0
PersistentKeepalive = 30

    Connexion du client

    Linux

    Placez le fichier de configuration wireguard à l'emplacement suivant : /etc/wireguard/aosc.conf

    Caution

    Ne pas utiliser de caractère spéciaux dans le nommage du fichier. Les interface virtuelle verra leur nom basé sur ce fichier, cela pourrait donc faire disfonctionner le processus de connexion.

    • Activation du tunnel VPN
    sudo wg-quick up aosc
    • Affichage du statut de la connexion
    sudo wg
    • Désactivation du tunnel VPN
    sudo wg-quick down aosc

    Info

    Pour une configuration automatique de la résolution DNS sur Linux, le paquet resolvconf doit être présent sur le system. Le fichier /etc/resolv.conf se doit d'être un lien symbolique du fichier /run/resolvconf/resolv.conf. Dans le cas contraire le changement de serveur DNS du système devra se faire manuellement.

    Gestion DNS

    Pour gérer les résolutions DNS du router PfSense, un script a été écrit pour permettre à tous de pouvoir disposer de la possibilité d'éditer ses propres résolutions.

    Ce script permet :

    • Lister toutes les entrées DNS sur serveur PfSense
    • Contrôler si une entrée (IP ou Hostname) est présente dans la base de données de PfSense.
    • Ajouter une enrée DNS
    • Supprimer une entrée DNS

    Dépendances :

    • os
    • requests
    • urllib3
    • click
    • json

    Pour satisfaire les dépendances :

    pip install ..

    L'outil est très simple d'utilisation.

    Example

    S'assurer que l'IP 10.0.2.10 n'a pas de résolution DNS qui pointe dessus : 

    ❯ python3 dns.py check -n dev-01
No match

    S'il y a match, alors les entrées concernées seront affichées : 

    ❯ python3 dns.py check -i 10.0.0.2
+----+---------+------------+----------+---------------------+
| ID |   Host  |   Domain   |    IP    |     Description     |
+----+---------+------------+----------+---------------------+
| 6  | proxmox | infra.aosc | 10.0.0.2 | Hyperviseur Proxmox |
| 9  | sandbox | infra.aosc | 10.0.0.2 | Hyperviseur Proxmox |
+----+---------+------------+----------+---------------------+

    Puis créer un nouvel enregistrement : 

    ❯ python3 dns.py create -i 10.0.2.21 -d casio.aosc -n dev-01 -c "Talos control-plane 1"
Request: Success

    Dans le cas ou le hostname ainsi que le domaine de la nouvelle entrée existerai déjà, le processus de création se stoppera pour ne pas créer de doubles entrées.

    Si besoin d'une suppression, il suffit de récupérer l'identifiant de l'alias pour le supprimer : 

    ❯ python3 dns.py remove 6
Request: Success

    Pour récupérer l'identifiant, il suffit de checker l'alias ou afficher la base de données entière : 

    ❯ python3 dns.py show
+----+----------------+------------+-----------+--------------------------------------+
| ID |      Host      |   Domain   |     IP    |             Description              |
+----+----------------+------------+-----------+--------------------------------------+
| 0  | casio-master-1 | casio.aosc | 10.0.2.11 |         casio talos master 1         |
| 1  | casio-master-2 | casio.aosc | 10.0.2.12 |         casio talos master 2         |
